# Diary Definition

This document clarifies what `posts/*/diary.md` is for.

The name `diary` can be misleading if read as a writing diary.
In this project, it should be understood as the team's working record.

---

## What A Diary Is

A `diary.md` file should preserve source reality close to the work itself.

Typical contents:

- production context
- observed behaviour
- deployment constraints
- debugging notes
- copied commands or logs
- partial interpretations at the time
- engineering responses that were actually tried
- phrases or fragments worth preserving from the real scene

It may be rough, incomplete, or repetitive.
That is acceptable.

---

## What A Diary Is Not

A `diary.md` file is not:

- a writing-plan file
- a publication strategy note
- an outline about how the article should be structured
- a place to compare candidate titles at length
- a place to debate article boundaries in meta-writing language

Those belong in `editorial/`.

---

## Practical Test

If a reader opens a diary and feels they are reading:

- what the team encountered
- what the system did
- what they tried
- what remained unresolved

then the file is probably shaped correctly.

If the reader instead feels they are reading:

- how the article should be written
- how two posts should be balanced
- what tone the final draft should adopt

then the file has drifted upward into editorial territory.

---

## Relationship To Post

`post.md` is the publishable layer.

`diary.md` is the source layer beneath it.

The post may become calm, shaped, and narratively controlled.
The diary can stay closer to raw working memory.

But both should still point toward system reality rather than writing-about-writing.
