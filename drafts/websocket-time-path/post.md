# 为什么同一个 `websocket URL` 会出现不同延迟

刚开始的时候，我以为他们不过是在做一次很普通的连接测试。

2022 年，团队在 Tokyo 部署一套 `CEX / DEX` 套利系统，需要同时订阅多路币安行情。对他们来说，这本来只是一次很普通的连接测试：同一个交易所、同一个 `region`、甚至同一个 `websocket URL`，同时建立多条连接，理论上这些连接的网络路径足够接近，延迟表现也不该差太多。

但连接建立后不久，连接之间就显示出延迟差异：其中一条始终快约 `1-2ms`，另外有几条则持续慢一些。

这种差异不是运行数小时后逐渐积累出来的，而是在连接建立后的早期阶段就已经出现，并在之后的运行中长期保持。

——

在他们开始讨论连接稳不稳的时候，延迟其实已经开始分层了。

但持续观察后会发现，它更像是一种结构性的时间分化。几条看起来连向同一个市场的 `websocket`，实际上像是在连接建立的瞬间，就已经落入了不同的时间轨道。

一个 `hostname` 往往并不对应单一入口。解析结果可能在多个 `IP` 之间变化，不同连接实例也可能被分配到不同的边缘节点或接入路径。即便部署环境完全一致，不同 `websocket` 也可能从基础设施的不同位置接入市场数据流。

但这并不能解释全部现象。

即使多个连接解析到相同的 `IP` 地址，它们的长期表现仍可能不同。更微妙的分化也可能发生在服务器端。行情系统通常通过有序的 `fan-out` 管线向外分发数据。连接一旦进入这一结构中的某个位置，后续消息的到达顺序和等待结构就可能在一段时间里保持相对稳定。延迟在这里不再只是传播时间，而更像是分发结构在时间上的一种投影。

——

从这个角度看，`websocket` 更像是一条时间路径，而不是单纯的数据通道。不同连接接收到的是同一市场的数据，但它们抵达系统的时间结构并不相同。

这件事真正重要的地方，不是“为什么偏偏这条更快”，而是实时系统会在连接建立的早期阶段，就被放进不同的时间结构里。看上去连接的是同一个市场，实际接住的却已经是不同的到达顺序、不同的等待位置、不同的延迟层次。
