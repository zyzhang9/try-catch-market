# 走海底光缆的流量：验证、原型与生产

## 问题

我们需要让特定目标的流量稳定地走海底光缆，而不是直接走本地默认公网出口。

这个需求不是单向的，也不只针对某一个站点。实际场景中，跨区域节点之间既可能需要把某些外部目标流量送到远端出口，也可能需要让回程流量同样经过海底光缆返回。因此，我们要解决的不是单点代理，而是一种可复用、可扩展、可进入生产环境的流量转发能力。

这件事的关键不只是“能不能转发”，而是以下几个问题：

- 是否足够稳定
- 是否对现有业务侵入最小
- 是否兼容 HTTP、HTTPS、WebSocket 等常见流量形态
- 是否适合长期运维，而不只是临时验证

围绕这个目标，我们评估了三种方案。

## 方案一：SSH 通道

第一种方案是在节点之间建立 SSH 通道，再由应用侧配置 `HTTP proxy` 或 `SOCKS proxy`，让目标请求通过 SSH 隧道转发。

### 优点

- 实现最快，适合快速验证链路是否可用
- 改动范围小，便于短时间内搭一个原型
- 对小规模测试比较方便

### 缺点

- 稳定性一般，不适合作为长期生产方案
- 本质上依赖应用层支持代理配置
- 对 WebSocket、长连接、自动重连等场景不够友好
- 如果应用、SDK 或连接库不支持代理，需要额外适配
- 故障排查通常停留在应用层，维护成本较高

### 判断

SSH 通道适合作为验证阶段的临时方案，但不适合作为正式生产方案。

## 方案二：Nginx Proxy

第二种方案是在中转节点部署 Nginx 作为代理入口，再由应用将目标请求通过 `HTTP proxy` 指向 Nginx。

### 优点

- 比 SSH 通道更工程化，便于集中管理
- 可以统一做连接转发和日志记录
- 适合标准 HTTP/HTTPS 请求的代理转发

### 缺点

- 仍然依赖应用层显式配置代理
- 对接入方式有要求，不是所有客户端和 SDK 都能自然支持
- WebSocket 虽然可以支持，但配置和运维复杂度更高
- 引入了额外的应用层代理组件，系统链路更长，故障点更多

### 判断

Nginx Proxy 比 SSH 通道更适合做原型，但仍然属于应用层代理方案，不适合作为统一的长期基础设施能力。

## 方案三：Route Table

第三种方案是在操作系统网络层直接控制流量路径，将目标地址的流量通过路由表定向到海底光缆对应的隧道或远端出口，由底层网络负责转发。

### 优点

- 对应用基本透明，通常不需要修改业务代码
- 不依赖 `HTTP proxy` 或 `SOCKS proxy`
- 更容易兼容 HTTPS、WebSocket、长连接等不同流量类型
- 网络路径更直接，架构更清晰
- 更适合长期生产环境运维
- 能把同一主机上的相关流量统一纳入控制

### 缺点

- 实施复杂度最高
- 需要同时处理隧道、路由、NAT、防火墙等网络组件
- 当目标域名的 IP 变化时，需要配套动态更新机制
- 排障要求运维人员具备较强的 Linux 网络基础

### 判断

Route Table 是三种方案里最复杂的一种，但也是唯一能够同时满足稳定性、通用性和低业务侵入的正式方案。

## 三种方案对比

| 方案 | 实现难度 | 稳定性 | 对应用改造 | WebSocket 兼容性 | 适合长期生产 |
| --- | --- | --- | --- | --- | --- |
| SSH 通道 | 低 | 低 | 高 | 一般 | 否 |
| Nginx Proxy | 中 | 中 | 高 | 中 | 一般 |
| Route Table | 高 | 高 | 低 | 高 | 是 |

## 为什么最终只能选第三个

综合比较后，我们最终只能选择第三种方案，也就是基于 Route Table 的流量分流。

原因很直接：

- 我们需要承载的不只是普通 HTTP 请求，还包括 HTTPS、WebSocket 和长连接场景
- 我们不希望把链路选择能力绑定在应用是否支持代理上
- 我们需要的是可复用的基础设施能力，而不是单个应用里的特殊适配
- 我们需要一套可以从验证阶段走到原型，再走到生产环境的长期方案

因此，正式方案必须在网络层完成目标流量的识别与转发，让符合条件的流量稳定走海底光缆。
