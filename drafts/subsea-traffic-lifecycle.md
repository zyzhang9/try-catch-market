# 走海底光缆的流量：验证、原型与生产

当我们说“让一部分流量走海底光缆”时，要解决的问题其实很简单：这些请求本来会从本地默认公网出口离开机器，现在我们希望它们改走另一条路径。比如，在 AWS London 和 AWS Tokyo 各放一台 EC2，在创建 VPC 时把 Avelacom 提供的接入配置挂上去，两边先实现互通，然后再决定哪些流量应该从这条链路上经过。

落到系统里，所谓“指定流量”往往也并不抽象，可能就是 `api.binance.com` 这样的 REST 请求，也可能是 `ws.kraken.com` 这样的 WebSocket 连接。

这个问题并不是单向的，也不是只针对某一个站点。东京侧可能希望访问某些交易所时从远端出口出去，比如把 `api.binance.com` 这类请求稳定送到 London 侧再出网；另一侧也可能需要让回程流量重新经过海底光缆带回来，比如让 `ws.kraken.com` 这类连接的路径尽量保持可控。

要建设的，并不是一个临时的代理技巧，而是一种可以逐步走到生产环境的流量控制能力。尤其当一条 London 到 Tokyo 的专线每个月已经在烧掉大约 3000 美元的时候，路径如何使用，已经不只是技术细节，而是系统设计本身的一部分。

一开始看，这似乎只是转发问题：流量能不能从另一侧出去，能不能回来，能不能连上目标服务。但只要真的把系统往前推进一步，问题就会立刻变得更具体。应用要不要改，WebSocket 能不能兼容，长连接稳不稳，运维上能不能长期承受，都会变成比“能不能通”更重要的判断标准。

如果从架构层次上看，这类方案大致可以分成三层。第一层是应用层，第二层是代理层，第三层是网络层。它们看起来是在提供不同方案，实际上更像是在回答不同阶段的问题。

最先出现的，通常是应用层方案。它最简单的实现方式，就是直接在两台机器之间搭一个 SSH 通道，再让应用去配置 HTTP proxy 或 SOCKS proxy，把目标请求送进这个通道。比如 London 上起一个出口节点，Tokyo 上的测试程序临时把访问 `api.binance.com` 的请求指到这个代理口。只要两端能 SSH 互通，原型几乎立刻就能搭出来。它很适合验证一件事，那就是“这条链路本身是不是可用”。

但应用层方案的边界也很快就会暴露出来。它依赖应用显式知道代理的存在，依赖 SDK 或连接库支持代理，依赖不同协议都能被这套接入方式覆盖。普通 HTTP 请求有时还算顺手，一旦进入 HTTPS、WebSocket、长连接或自动重连场景，复杂度就会迅速上升。于是应用层方案更像一种验证方法，而不是长期结构。它能够帮助我们确认海底光缆这条路是通的，却不适合承载生产系统的稳定预期。

再往前一步，就会进入代理层。代理层最简单的实现方式，是在中转节点上部署一个 Nginx Proxy，再由应用把目标请求送到这个代理入口。比如把 Nginx 放在 London 的那台机器上，让 Tokyo 侧的应用统一把 `api.binance.com` 这类目标域名的请求指过去。

和 SSH 通道相比，这已经不是纯粹的临时技巧，而更像一个正式组件了。流量先进入代理，再由代理统一向外转发，系统也终于有了一个可以集中管理、集中观察的入口。

代理层比应用层更进一步的地方，在于它开始具备“系统感”。日志可以集中，入口可以固定，行为也更容易被观察。作为原型，它往往比 SSH 通道更像一个能被持续使用的东西。

但它仍然保留了应用层最核心的问题：应用依然需要知道代理的存在。也就是说，虽然转发这件事被从 SSH 通道换成了 Nginx Proxy，但应用仍然要改，客户端仍然要配，接入方式仍然依赖不同协议和不同运行时的支持。WebSocket 当然不是不能做，而是做起来会明显更复杂。于是代理层虽然比应用层更成熟，却依然停留在“原型可行”的阶段，还没有真正进入“生产可依赖”的状态。

——

走到第三层，问题的形状才会发生变化。网络层最简单的实现，不再是让应用知道代理，而是在主机上建立一条通往海底光缆出口的隧道，然后通过 route table 把目标流量直接送进这条隧道。比如在 Tokyo 和 London 之间先建一条隧道，再把 `api.binance.com`、`ws.kraken.com` 这类目标域名解析出来的 IP 定向路由到 London 侧出口。

到了这一层，应用不再负责决定路径，它只负责发起正常请求，真正决定“这包流量从哪里走”的，是底层网络本身。

这也是为什么网络层方案和前两层有本质区别。应用层和代理层都在尝试让应用配合网络，而网络层开始让网络自己承担路径控制。HTTP、HTTPS、WebSocket、长连接，在这里不再是几种需要分别适配的接入形态，而更像是同一条路径上的不同负载。对于业务系统来说，这种透明性本身就是最大的价值。

当然，网络层也最复杂。哪怕只看最简单实现，也已经离不开隧道、路由、NAT、防火墙和动态更新这些问题。目标域名不是固定 IP，路由规则却只能基于 IP 工作；一条路径可以先建起来，但它是否能长期稳定工作，又取决于一整套配套机制是否跟得上。和前两层相比，这里几乎没有捷径可言。

但正因为没有捷径，它才真正接近生产。应用层解决的是“先跑起来”，代理层解决的是“先管起来”，网络层解决的则是“真正用起来”。这三层并不是互相排斥，更像是一种自然推进：先验证，再原型，最后进入生产。

——

如果把这三种方案放在一起比较，结论其实并不复杂。应用层最容易起步，适合快速验证链路；代理层更像一个中间形态，适合做可管理的原型；网络层门槛最高，却也是唯一真正能够把“指定流量走海底光缆”做成长期能力的方案。

所以最后的判断并不难。只要目标还是验证，应用层已经足够；如果只是想做出一个更像系统的实验版本，代理层也完全合理；但如果目标是生产，那么最后只能落到第三层，也就是网络层。

原因很简单。我们要承载的不只是普通 HTTP 请求，还包括 HTTPS、WebSocket 和长连接。我们不希望把链路选择能力散落在不同应用、不同 SDK、不同连接库里，也不希望每一类流量都重新做一遍接入适配。适合作为基础设施能力的，只能是让网络在系统层面接管路径控制。

于是，海底光缆流量方案的验证，可以从应用层开始；原型，可以在代理层停留；但生产，最终必须进入网络层。
