# Notes

## Context

- 目标是让一部分流量走海底光缆，而不是本地默认公网出口
- 例子：在 AWS London 和 AWS Tokyo 各放一台 `EC2`
- 在创建 `VPC` 时挂上 Avelacom 提供的接入配置
- 两边先实现互通，再决定哪些流量走这条链路

## Target Traffic

- 指定流量不抽象
- 可能是 `api.binance.com` 这样的 `REST` 请求
- 也可能是 `ws.kraken.com` 这样的 `WebSocket` 连接
- 问题不是单向的，也不只针对一个站点
- Tokyo 侧可能希望某些请求从 London 侧出口出去
- 另一侧也可能希望回程流量重新经过海底光缆带回来

## Why It Matters

- 目标不是临时代理技巧，而是能逐步走到生产环境的流量控制能力
- London 到 Tokyo 的专线每个月大约 `3000` 美元
- 路径如何使用已经是系统设计的一部分

## Layer 1: Application

- 直接在两台机器之间搭 `SSH` 通道
- 应用显式配置 `HTTP proxy` 或 `SOCKS proxy`
- Tokyo 上的测试程序把目标请求送进代理口
- 适合快速验证链路是否可用
- 边界：
  - 应用显式知道代理存在
  - `SDK` / 连接库要支持代理
  - 不同协议要分别适配
  - 进入 `HTTPS`、`WebSocket`、长连接、自动重连后复杂度上升
- 更像验证方法，不适合承载生产稳定预期

## Layer 2: Proxy

- 在中转节点部署 `Nginx Proxy`
- 应用把目标请求送到代理入口
- 比 `SSH` 通道更像正式组件
- 流量入口固定，可集中管理和观察
- 适合作为更像系统的原型
- 边界：
  - 应用仍然知道代理存在
  - 应用仍然要改
  - 接入方式仍依赖协议和运行时支持
  - `WebSocket` 可以做，但复杂
- 仍停留在“原型可行”

## Layer 3: Network

- 在主机上建立通往海底光缆出口的隧道
- 通过 `route table` 把目标流量直接送进隧道
- 在 Tokyo 和 London 之间先建隧道
- 再把 `api.binance.com`、`ws.kraken.com` 这类目标域名解析出的 `IP` 定向路由到 London 侧出口
- 应用不再负责决定路径
- 真正决定流量从哪里走的是底层网络
- 价值：
  - `HTTP` / `HTTPS` / `WebSocket` / 长连接都变成同一路径上的不同负载
  - 业务系统获得透明性
- 代价：
  - 隧道
  - 路由
  - `NAT`
  - 防火墙
  - 动态更新
  - 域名不是固定 `IP`，路由规则却只能基于 `IP`
- 没有捷径，但真正接近生产

## Working Conclusion

- 应用层适合验证链路
- 代理层适合做可管理原型
- 网络层才适合把“指定流量走海底光缆”做成长期能力
- 核心原因：不希望把链路选择能力散落在应用、`SDK`、连接库里
- 适合作为基础设施能力的，只能是让网络在系统层面接管路径控制
