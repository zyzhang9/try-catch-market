# 他们不是一开始就自己写 driver

刚开始的时候，他们并没有打算自己把整套交易所接入都写一遍。

那时系统还在很早的阶段，很多东西甚至还不能算系统。更像几段刚接上的链路：先连上 Binance，先把行情拿回来，先能查余额，先能下一张单，先让一个最小实盘链路真的跑起来。那种时候如果还坚持从 `REST` 到 `websocket` 全部自己写，事情通常会慢得不太像工程推进，反而像在提前偿还一笔还没有发生的债。

所以他们先用了 CCXT。

这一步没什么问题。回头看，甚至很难说还有比这更合适的起点。统一接口让他们可以先把交易所接起来，现货和合约至少先有一层相似的调用方式，原型、toy、回测外壳，也都能借着这层接口比较快地长出来。最开始的目标本来就不是“定义最终系统语义”，而是先让系统和外部市场之间有一条能工作的线。

——

真正的变化不是突然发生的。

起初业务代码里到处都能直接碰到 CCXT。查余额是一层，查持仓是一层，下单撤单又是一层。看上去还算顺手，因为系统规模还小，很多问题也还没有真正冒出来。那时候他们面对的更多是“有没有接通”，而不是“这条路径以后的语义到底归谁定义”。

等进程开始常驻，连接开始反复重连，订单回报开始进入本地状态机以后，事情就慢慢变了。最早出现的不适感，通常并不来自 `fetch_balance` 或 `create_order` 这类接口。前期这些接口反而很稳定，也足够有用。真正先长出差异的，多半是实时事件：`websocket` 怎么补，用户数据流断了以后从哪一段接，订单回报晚到了几百毫秒以后，本地该先相信成交、余额，还是持仓。

他们没有立刻把 CCXT 扔掉。

更常见的动作，是先在外面包一层自己的 adapter。不是因为那一刻他们已经想好最终架构了，而是业务代码开始不愿意再直接碰底层库。团队慢慢发现，只要业务逻辑在各处直接调用统一接口，后面很多本来只属于接入层的问题，都会直接渗进系统内部。一个字段名怎么解释，一种异常怎么重试，一次返回不完整时该不该信，都会在不同角落各自长出一份。

他们还没有把这件事叫作“分层”，只是开始本能地把边界往回收。

——

那层 adapter 最开始也很朴素。它未必比 CCXT 本身复杂多少，只是先把“业务想做什么”和“交易所接口长什么样”隔开一点。业务代码只看自己的下单语义、自己的账户对象、自己的错误类型，底下究竟是 CCXT 哪个方法，返回值里哪个字段需要修一下，先不让上层知道。

这一步其实已经很重要了。因为系统第一次承认，统一库给出来的对象模型未必就是自己以后想长期相信的模型。很多时候，变化不是从“我要重写整个 driver”开始的，而是从“不要再让业务代码直接认识 CCXT”开始的。

再往后，替换也不是平均发生的。

他们不会先把所有 `REST` 都改写掉。因为问题并不平均分布。查询接口有时候还可以继续留给 CCXT，甚至行情里的某些简单查询也还够用。真正越来越难继续交给统一层的，是那些一旦出错就会直接改写系统状态的地方。`websocket` 是一层，订单生命周期是一层，`position / balance sync` 是一层，`retry / reconnect / idempotency` 又是一层。再往下，不同交易所各自的异常、字段缺口和回报顺序，也都开始逼着系统把解释权收回来。

——

他们后来处理这件事的方式，不像一次迁移，更像一种局部收口。

先是 `websocket` 自己写。因为连接一断，后面所有东西都会跟着变形。补快照、补缺口、恢复订阅、确认本地状态有没有在断线窗口里漂掉，这些问题一旦真的进入生产，就很难再继续停留在“统一事件流”里。再后来是订单生命周期。下单、撤单、部分成交、重试、超时以后怎么标记本地状态，慢慢都需要由系统自己来定义。到了这一步，driver 写不写其实已经不太像一个风格选择，更像是执行路径被迫长成了自己的形状。

所以最后留下来的，通常不是“全面去 CCXT”，而是一种更别扭也更真实的结构。查询层有些地方还在用它，某些不那么敏感的接入也还留着；但关键路径已经不再信任那套统一语义。系统自己维护订单状态机，自己维护账户事件流，自己决定断线以后从哪里恢复，也自己记录某个交易所上的特殊例外到底该怎么算。

到了这个阶段，CCXT 还在，只是位置已经往下沉了。它更像接入材料，而不是一层可以直接承接生产语义的系统边界。

我后来很少再听见他们讨论“要不要全自研”。他们讨论的是另一类问题：下一层该不该收回来，哪一段路径已经不能继续借别人的定义往前走。这个问题听起来没那么整齐，但更接近现场里真正发生过的事。

——

我是 Drift，我看过他们先把东西接起来，再一点点把关键路径收回自己手里。
