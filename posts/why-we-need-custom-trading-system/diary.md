# Notes

## Context

- 2019 年在北京做加密钱包公司的后端交易系统
- 入口已经不是策略，而是钱包用户的真实交易需求
- 公司自己不做内部撮合，订单要路由到外部交易所执行
- 系统形状从 `strategy-driven` 慢慢变成 `flow-driven`

## Execution Shape

- 同时连多家交易所
- 同时维护多条 `websocket` 和 `REST` 连接
- 同一笔需求可能沿不同路径执行
- 某条连接只要慢几十毫秒，报价和路由就开始偏
- 某一家交易所不可用，流量会立刻重新分配到别处
- 流量一重分配，库存、成交率和手续费会一起变形

## Framework Tension

- 大部分开源框架默认单执行路径
- 即使支持多个 `broker` 或多个 adapter，也更像并排接入
- 不太自然表达多路径执行系统
- 问题不再只是“订单怎么发出去”
- 更像“这一笔需求现在该沿哪条路径完成”

## CCXT Reality

- 最早接交易所接口时大量使用 CCXT
- 统一 API 确实让系统先接了起来
- 现货和合约至少先有一层相似调用方式
- 真正跑起来以后问题慢慢出现，尤其是 Binance futures
- 某些字段和交易所原始语义对不上
- 某些状态更新比真实回报晚半拍
- `order`、`position`、`balance` 不总在一个时钟上

## Fork Consequence

- 后来把 CCXT 从 GitHub fork 下来，本地修改，再提 PR
- PR 很久没有回应，本地版本却不能停
- 上游继续变化，业务系统也继续往前走
- 项目里开始同时维护两套东西：
  - 上游统一抽象
  - 本地为了生产接住现实而留下来的 patch
- 调试时经常先问：
  - 这是 CCXT 原来的行为
  - 还是本地 patch 造成的
- 库最后更像原材料，而不是现成答案

## Exchange Timing

- 同样是成交和余额更新，不同交易所顺序并不一样
- Binance 经常先收到 `trade execution`，再收到余额更新
- Kraken 经常先看到余额变化，成交明细稍后才到
- 这个时间差有时几十毫秒，有时几百毫秒
- 放在日志里不长，放在本地状态机里足够让仓位、余额、可用资金暂时分开

## Local Consequence

- 如果统一先信成交：
  - 某些交易所上仓位先变，资金侧还停在旧值
- 如果统一先信余额：
  - 另一类交易所上资金先动，成交回报却还没到
- 问题不是最终会不会补齐
- 问题是很多决定都发生在这段不一致窗口里：
  - 是否继续补单
  - 是否撤掉旧单
  - 是否认定本地仓位超限
  - 是否把流量切到别处

## Working Direction

- 系统最后只能把越来越多的交易所差异直接写进去
- Binance 和 Kraken 不再共用同一套事件顺序假设
- 不同连接的健康度不再只是监控数字
- 路由也不再只是“有没有更好价格”
- 系统越来越像一张不断扩展的交易所差异表
- 上面会记：
  - 回报顺序
  - 接口缺口
  - `partial fill` 该先改仓位还是先改余额
  - 哪类异常回报应该先挂起等下一条消息来补

  - 而是从 `ccxt` 起步，再慢慢把关键路径收回来
